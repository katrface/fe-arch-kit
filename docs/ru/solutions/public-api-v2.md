# Точка входа в модуль (Public API)

## Суть паттерна
В корне папки создается файл-шлюз (обычно `index.ts`), который является единственным законным способом взаимодействия с содержимым этого модуля.

*   **Файл-фасад:** В нем перечисляются только те компоненты, функции или типы, которые модуль готов предоставить остальному приложению.
*   **Приватность по умолчанию:** Всё, что не экспортировано через этот файл, считается «секретным» и не может быть использовано снаружи.

## Правила использования
1.  **Запрет «глубоких» импортов:** Нельзя импортировать файлы напрямую из недр модуля (например, `import { Button } from './features/auth/ui/Button'`).
2.  **Обращение к корню:** Любой импорт должен заканчиваться на названии папки модуля (`import { AuthForm } from '@/features/auth'`).
3.  **Явный экспорт:** Вместо автоматического `export *`, в `index.ts` лучше прописывать экспорты осознанно, чтобы не выставить наружу лишние вспомогательные утилиты.

## Получаемые свойства

### 1. Гибкость рефакторинга
Вы можете полностью перестроить внутреннюю структуру модуля (переименовать файлы, разбить компоненты, изменить вложенность). Пока `index.ts` отдает те же сущности, внешние потребители кода не заметят изменений и ничего не сломается.

### 2. Снижение сложности (Инкапсуляция)
Разработчику не нужно изучать 20 файлов внутри чужого модуля, чтобы понять, как его использовать. Достаточно открыть один «фасад», который служит краткой документацией возможностей модуля.

### 3. Защита от «спагетти-кода»
Паттерн ограничивает количество связей между частями системы. Модуль становится «черным ящиком», что предотвращает случайное запутывание логики, когда внутренние хелперы одного модуля внезапно начинают использоваться во всем проекте.

### 4. Стабильность API
Выделение Public API заставляет разработчика модуля задуматься: «Что из этого действительно должно быть доступно другим?». Это ведет к проектированию более чистых и стабильных интерфейсов.

## Роль линтера (Автоматизация)
Паттерн не работает на одних «честных словах». В проекте должны быть настроены правила (например, через ESLint), которые решают две задачи:

1.  **Идентификация:** Определяют, какие именно папки в проекте считаются модулями и обязаны иметь точку входа.
2.  **Запрет:** Блокируют любые попытки импорта из подпапок модуля, если запрос идет извне.

Public API — это паттерн организации кода, при котором взаимодействие с модулем (папкой) происходит строго через единый интерфейс — файл-шлюз (обычно index.ts). Все внутренние файлы модуля считаются приватными деталями реализации.

Суть по фактам:
- **Файл-фасад:** В корне папки создается index.ts, который содержит именованные экспортируемые элементы (компоненты, функции, типы).
- **Инкапсуляция:** Остальные файлы в поддереве «спрятаны».
- **Единый контракт:** Внешний мир видит не файловую структуру модуля, а только то, что модуль сам решил «показать».

Правила и ограничения
Запрет глубоких импортов: Запрещено импортировать что-либо напрямую из внутренних файлов (например, import { User } from './module/ui/User.tsx').
Доступ только через шлюз: Любое использование содержимого модуля должно идти через его точку входа (import { User } from './module').
Осознанный экспорт: Не рекомендуется использовать export * from.... Каждый экспорт должен быть явно прописан, чтобы не выпустить наружу лишние детали.
