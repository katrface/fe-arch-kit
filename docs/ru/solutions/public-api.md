# Public API

## Семантика
Создать единую точку входа в модуль (поддерево файлов), чтобы описывать/контролировать предоставляемый интерфейс (контракт) данного модуля, делая из него чёрный ящик и скрывая внутреннюю сложность.

Необходима поддержка правилами eslint.

#### Требования
- Запретить импорты в поддерево файлов из-вне в обход `index.ts`
- Использовать явный экспорт, избегая `export *` в `index.ts`

#### Рекомендации
- Для элементов указанных в Public API ужесточить требования к надежности и качеству автотестов
- Можете создавать несколько точек входа в модуль для избегания циклических зависимостей, но важно придерживаться семантики

#### Свойства
- Гибкость в рефакторинге, не затрагивая интерфейсы содержимого в Public API
- Наглядность и контроль

#### Риски
- Создается код, не несущий функциональной ценности, сам по себе

## Когда использовать
- Реализуете модьную архитектуру

## Синтаксис

Структура папок:
```
.../
├── module-1/
│   ├── module-1-1/
│   │   ├── ...
│   │   └── index.ts // Public API
│   ├── ...
│   └── index.ts // Public API
├── module-2/
│   ├── ...
│   └── index.ts // Public API
└── ...
```

## Автоматизация
1) Определить где должен использоваться Public API
2) Запретить импортов в обход `index.ts`
